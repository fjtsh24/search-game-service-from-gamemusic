# 02 設計上の重要な問いと決定事項

企画段階で生じた設計上の分岐点と、その決定内容・理由を記録する。

---

## 決定①：「気付き」の提供方法

### 問い

ペルソナBに「自分がBGMでゲームを選んでいた」という気付きを与えるにあたり、  
どのタイミング・どの形で提供するか？

### 検討した案

| 案 | 内容 |
|----|------|
| 案X | ライブラリ連携時に「あなたの好みはこうです」と分析結果を提示（オンボーディング体験） |
| 案Y | プロフィールページに音楽趣味の分析を常時表示（じわじわ気づかせる） |
| **案Z（採用）** | レコメンドを通じて自然に気付かせる。「好みを教える」のではなく「新しいゲームの提案」の中で気付きを与える |

### 決定内容

**案Zを採用。**

- 好みを押し付ける形は取らない
- Last.fmの「テイストの似たアーティスト」のように「こういうゲームはどうですか？」と提案する
- 提案が合っていたかどうかのフィードバックを軸に、ユーザー自身が自分の好みに気付く
- 気付きは機能ではなく、良いレコメンドの副産物として生まれる

### 設計への含意

- 「あなたの音楽的好みはXです」と表示する機能は不要
- レコメンドの理由表示（「〇〇が好きな人に人気」など）が気付きの補助になる
- フィードバックループ（合った/合わなかった）がサービスの学習とユーザーの気付き両方を担う

## 決定②：「音楽の類似性」の構築方法

### 問い

サービスの根幹となる「音楽的に似たゲーム」をどう定義・構築するか？

### 検討した案

| 案 | 内容 | MVP採否 |
|----|------|---------|
| **案α** | 既存の音楽グラフを活用（Spotify/Last.fm APIでアーティスト類似度を取得し、ゲームに紐付け） | ✅ MVP採用 |
| **案β** | タグ・ムードをユーザーが付与してグラフを育てる（コミュニティ編集） | ✅ MVP採用 |
| 案γ | プレイ履歴とゲームの音楽メタデータから機械学習で導出 | 🔜 将来拡張 |

### 決定内容

**MVP：案α + 案β の組み合わせ**  
**将来：案γ（データが蓄積されてから）**

### 活用候補のデータソース

| データソース | 用途 | 採否 |
|------------|------|------|
| Steam Soundtracks カテゴリ | ゲームデータの主ソース（人気順取得） | ✅ 採用 |
| MusicBrainz API | 作曲家名の補完（Steam インポート内で使用） | ✅ 補完用途のみ採用 |
| Last.fm API | アーティスト間の類似度・タグ | ✅ 採用 |
| IGDB | ゲームメタデータ・カバー画像 | 🔜 MVP後検討 |
| Steam API | ユーザーのゲームライブラリ取得 | ✅ 採用 |
| VGMdb | ゲーム × 曲 × 作曲家の紐付け | ❌ 非公式APIのため採用見送り |
| Spotify API | アーティスト類似度・楽曲特徴量 | ❌ 25ユーザー制限のため採用見送り |

## 決定③：音楽の好みの表現方法

### 問い

「音楽の好み」をどう表現するか？

### 検討した案

| 案 | 内容 |
|----|------|
| A | 楽曲・アーティスト単位（具体的な好みから探す） |
| B | タグ・雰囲気単位（言葉で絞り込む） |
| **C（採用）** | 両方を組み合わせる |

### 決定内容

**案Cを採用。Last.fmのイメージに近い形。**

- アーティスト/楽曲を起点にして、そこからタグで広げる
- 具体的な入口（アーティスト検索）と抽象的な入口（タグ・雰囲気検索）の両方を持つ
