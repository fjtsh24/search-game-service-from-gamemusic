# 01 ジョブ分析・ペルソナ・競合マップ

## 1. ジョブ理論（Jobs-to-be-Done）分析

### 検討した仮説

企画初期に3つの利用シーンを仮説として検討した。

| 仮説 | 内容 | 採否 |
|------|------|------|
| **仮説A** | 好きなゲームのサントラを聴いていて「このアーティストの他の作品が入ったゲームをやりたい」と思った瞬間に使う | ✅ 採用（副ユースケース） |
| **仮説B** | 次に遊ぶゲームを探したいが、ジャンルや評価では分からない。音楽の雰囲気で絞り込みたい | ✅ 採用（主ユースケース） |
| **仮説C** | 自分の音楽趣味に近い人がどんなゲームをやっているか知りたい | ❌ 不採用（スコープ外） |

### 決定したコアジョブ

> **「ジャンルや評価では伝わらない"ゲームの雰囲気"を、音楽の好みという軸で探せるようにする」**

仮説BがメインユースケースでAがサブ。仮説Cはスコープから外す。

## 2. ペルソナ定義

### ペルソナA：音楽起点の探索者

| 項目 | 内容 |
|------|------|
| **特徴** | 音楽をよく聴き、ゲーム音楽にも詳しい |
| **行動** | Spotifyで好きな音楽を掘り、Last.fmでアーティストを辿るような行動が日常 |
| **欲求** | 「このアーティストのゲームをやってみたい」 |
| **サービスへの入り方** | アーティスト・楽曲名で検索してサービスに入る |
| **対応ユースケース** | 仮説A・仮説B両方 |

### ペルソナB：無自覚な音楽感受性ゲーマー

| 項目 | 内容 |
|------|------|
| **特徴** | ゲームをよく遊ぶ。好きなゲームに共通点がある（実は音楽）だが気付いていない |
| **行動** | 「なんか刺さるゲームとそうでないゲームがある」と感じている |
| **欲求** | 良いゲームをもっと見つけたい（音楽軸であることは自覚していない） |
| **サービスへの入り方** | ゲームライブラリ登録・レコメンド機能から価値を得る |
| **対応ユースケース** | 仮説B |
| **このサービスが与えたいもの** | 「自分がBGMで選んでいた」という気付き（押し付けず、レコメンドを通じて自然に） |

## 3. 競合マップ

| サービス | 音楽発見 | ゲーム発見 | 音楽→ゲーム連携 | 備考 |
|---------|---------|---------|--------------|------|
| Last.fm | ◎ | ✗ | ✗ | 音楽探索体験のリファレンス |
| VGMdb | △ | △ | △ | データベースとしては有用。探索・レコメンド機能なし |
| Spotify / Apple Music | ◎ | ✗ | ✗ | ゲームとの接続なし |
| Steam | ✗ | ○ | ✗ | ジャンル・評価軸のみ |
| RateYourMusic | ○ | △ | ✗ | ゲームサントラも対象だが発見機能は弱い |
| BackLoggd / HowLongToBeat | ✗ | ○ | ✗ | ゲーム管理系。音楽軸なし |

### 競合ギャップ（このサービスが埋める空白）

**音楽の好みを起点にゲームを発見できるサービスが存在しない。**

- VGMdbはゲームと音楽のリンクを持つが、探索・レコメンド機能が弱い
- Last.fmのような「テイストを辿る体験」がゲーム発見に存在しない
- Steamなどのゲーム発見は評価・ジャンル軸に偏っている
